# AI Agent工具设计：Claude Code构建经验总结

**来源：**[AI Agent工具设计：Claude Code构建经验总结](https://x.com/i/article/2027446899310313472)

这篇文章总结了构建Claude Code过程中关于AI代理工具设计的核心经验，。其核心观点是：工具设计需要与代理的“能力形状”相匹配，这是一门依赖具体模型、目标和环境的艺术，而非纯科学。

## 提问能力的演进
提升代理的提问能力（elicitation）经历了多次迭代。最初团队尝试在ExitPlanTool中增加问题参数，但代理会困惑于同时输出计划和相关问题。随后改用修改后的Markdown格式输出问题，虽然代理能较好遵守，但格式不稳定，可能漏掉选项或追加多余句子。最终，团队开发了专用的AskUserQuestion工具，允许代理在任意时刻调用，通过弹窗展示结构化问题并阻塞循环直到用户回答。这个工具不仅稳定了输出格式、支持多选项呈现，更重要的是代理“愿意”调用它，实现了低摩擦、高带宽的用户沟通。

## 工具随能力变化
随着代理能力提升，工具需求也在演变。早期代理需要Todo列表来保持执行轨道，通过TodoWrite工具写入和更新，甚至依赖每5轮的系统提醒防止偏离。但当代理能力增强后，持续提醒反而可能限制其主动性，误以为必须死守原列表。同时，多子代理协作需要更复杂的协调机制。于是团队将TodoWrite替换为Task工具：Todo聚焦于“让代理不跑偏”，而Task支持依赖关系管理、子代理间共享更新，允许代理动态修改任务，更好地适应多代理协作场景。这体现了工具设计需要随模型能力演进，过去“必需”的工具可能成为今天的限制。

## 搜索与上下文构建
代理在获取上下文方面的能力显著进化。初期依赖RAG向量数据库被动接收上下文，虽快但需要索引初始化，且上下文是“喂给”代理的。突破来自于提供Grep工具，使代理能在代码库中主动搜索，自行构建上下文。随着代理变聪明，只要给予合适工具，它越来越擅长自主构建上下文。

这一进展促成了“渐进披露”（Progressive Disclosure）理念的正式化，通过Agent Skills机制实现。代理可读取skill文件，skill文件又能引用其他文件，形成递归搜索。skills常被用来扩展代理的搜索能力，例如告诉它如何调用某个API或查询数据库。在一年时间里，代理从“几乎无法自主构建上下文”演进到“能跨越多层文件做嵌套搜索，并精确找到所需上下文”。

## 渐进披露实践
针对代理对自身用法知识不足的问题，团队尝试了渐进披露方案。将全部知识塞入system prompt会增加上下文负担，干扰主要编码任务。提供文档入口链接让代理需要时加载检索虽然有效，但代理常将大量检索结果塞入上下文，而用户往往只需要最终答案。于是团队构建了Claude Code Guide子代理：当用户询问Claude Code自身用法时，主代理被提示调用这个子代理。该子代理拥有更完整的“如何高效检索文档、如何组织返回结果”的专门指令，在扩展动作空间的同时避免了新增工具。

## 设计原则
从这些经验中提炼出若干设计原则。工具设计需基于代理的实际能力，通过观察和实验了解其“能力形状”。工具需要随代理能力提升而迭代更新，过去“必需”的工具可能成为限制。渐进披露通过分层、递归的机制让代理按需发现上下文，避免一次性加载所有信息。结构化交互通过专用工具稳定输出格式，降低用户交互摩擦，提高沟通效率。多代理场景需要工具支持依赖管理、状态共享和动态任务调整。

## 结论
AI代理工具设计是一个动态的、观察驱动的过程。成功的关键在于“学会像代理一样思考”：理解代理如何感知问题、需要什么工具来解决问题，并据此设计与之匹配的动作空间。持续实验、阅读输出、尝试新方法是必不可少的实践。工具设计最终是艺术与科学的结合，需在模型能力、目标与环境之间找到最佳平衡。这门艺术的核心在于多做实验，持续读输出，反复尝试新方法，真正学会像Agent一样去看问题。
